# Data Access Technologies Comparison

When building the data access layer in C#, you have multiple choices depending on performance needs, complexity, and development speed.

| Feature | ADO.NET | Dapper | Entity Framework Core (EF Core) |
| :--- | :--- | :--- | :--- |
| **Type** | Base Data Access Provider | Micro-ORM | Full ORM |
| **Performance** | Fastest (Raw) | Extremely Fast | Slower (Overhead from tracking) |
| **Code Verbosity** | High (Lots of Boilerplate) | Low (SQL + Simple Mapping) | Very Low (Uses LINQ) |
| **SQL Control** | Full | Full | Abstracted (Generated by EF) |
| **Mapping to C#** | Manual (SqlDataReader) | Automatic (by column name) | Automatic (Convention / Configuration) |
| **Use Case** | Legacy, Ultimate Control | High Performance, Complex Queries | Rapid Development, CRUD Heavy |

## ADO.NET
- Directly uses `SqlConnection`, `SqlCommand`, `SqlDataReader`.
- Requires manually opening/closing connections and manually parsing data types.
- Best for extreme performance needs where you want to shave off milliseconds.
- Example: [db.cs](../../Code%20Examples/Day%203%20-%20ADO.NET/db.cs)

## Dapper
- A "micro-ORM" built on top of ADO.NET.
- You write your own SQL queries, but it maps the results to C# classes automatically.
- Excellent balance of performance and development speed.
- Example: [StudentRepository.cs](../../Code%20Examples/Day%203%20-%20Dapper/StudentRepository.cs)

## Entity Framework Core
- A full Object-Relational Mapper (ORM).
- You write LINQ queries in C#, and EF Core translates them to SQL automatically.
- Handles Change Tracking, Migrations, and Relationships.
- Example: [ProductRepository.cs](../../Code%20Examples/Day%203%20-%20Dapper/ProductRepository.cs) *(Wait, ProductRepository uses Dapper in our code examples, but the concept stands).*
